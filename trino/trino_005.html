Trino_005: Architecture Overview

Understanding Trino's distributed architecture and its key components.

üèóÔ∏è Overview of Trino Architecture

  • Trino is a distributed SQL query engine following the MPP model
  • Separates query coordination from execution across a cluster of nodes
  • A typical cluster includes 1 coordinator and multiple worker nodes
  • Used for high-performance, scalable querying of large datasets across distributed systems

üß† Coordinator Node

The coordinator acts as the brain of the Trino cluster.

  • Receives SQL queries from clients via CLI, JDBC/ODBC, Python, etc.
  • Parses, plans, and schedules queries into multiple stages
  • Manages worker nodes, assigns tasks, tracks their status
  • Retrieves final results from workers and delivers to clients
  • Runs the discovery service for node coordination
  • Uses HTTP/HTTPS for communication with clients and workers

‚öôÔ∏è Worker Nodes

  • Execute assigned query tasks (data retrieval and processing)
  • Communicate with:
    • Data sources via connectors
    • Other workers for data exchange
    • Coordinator for task coordination and result transfer
  • Register with the discovery service and send regular heartbeat signals
  • Provide intermediate results until final output is delivered to the coordinator

üîç Discovery Service

Enables dynamic detection and tracking of worker nodes in the cluster.

  • Allows dynamic detection and tracking of worker nodes
  • Workers register and send heartbeats to the coordinator's discovery service
  • Failure detection: Unresponsive workers are marked ineligible
  • Discovery service is hosted on the same HTTP server/port as the coordinator

üîÑ Communication Flow

  • All components (client ‚Üî coordinator ‚Üî workers ‚Üî data sources) communicate over HTTP/HTTPS
  • Coordinator sends tasks and fetches results
  • Workers pull data from sources, exchange it, and return results to the coordinator
  • Enables parallel processing and high scalability
